# LTC 项目全局执行要求

## 数据库表与字段注释

- 创建数据库表时，必须为表增加明确的中文 `COMMENT` 注释。
- 表中的每一个字段都必须增加明确的中文 `COMMENT` 注释，包括主键、状态、版本号、创建人、创建时间、更新人和更新时间等技术字段。
- 字段注释应优先使用与前端页面一致的业务名称，并补充必要的编码含义、单位或取值范围，便于通过数据库元数据快速匹配前端字段。
- 禁止使用“字段1”“备用字段”“相关信息”等无法体现业务含义的模糊注释。
- 修改已有表或字段时，如果注释缺失或业务含义已经变化，必须通过新的数据库迁移补充或修订；不得修改已经执行过的迁移文件。
- 完成数据库变更前，必须查询 `information_schema.TABLES` 和 `information_schema.COLUMNS`，确认本次涉及的表及字段不存在空注释。

## 前后端开发安全要求

- 前后端设计、编码、依赖安装、测试和交付必须将安全作为验收条件，不得为了快速实现而跳过必要的安全控制。
- 选择或升级框架、运行时、数据库驱动、构建工具和第三方依赖前，必须检查官方安全公告、CVE 信息和当前依赖审计结果；不得主动引入已公开且确认受影响的高危或严重漏洞版本。
- 依赖必须通过锁文件固定版本。交付前必须执行适用于当前技术栈的依赖安全审计，并处理高危、严重漏洞；如果暂时没有安全版本或无法修复，必须停止交付并明确报告风险、影响范围和临时缓解措施，禁止隐瞒。
- 所有身份认证、角色权限、数据权限和业务状态权限必须由后端强制校验，不能只依赖前端隐藏菜单、按钮或字段。
- 所有外部输入都必须在后端进行类型、长度、格式、取值范围和业务规则校验；数据库访问必须使用参数化查询或安全的数据访问层，禁止拼接用户输入生成 SQL。
- 前端不得直接渲染未经可信清洗的 HTML，不得使用不安全的动态代码执行方式；必须防范 XSS、CSRF、开放重定向、敏感信息泄露和越权访问。
- 密码、令牌、数据库连接密码、私钥等敏感信息不得写入前端代码、源码、日志、测试数据或 Git 仓库，必须通过环境变量或专用密钥管理机制提供，并确保日志进行脱敏。
- 文件上传必须限制文件数量、大小、扩展名和实际内容类型，使用服务器生成的随机文件名，阻止路径穿越、脚本文件执行和未授权下载；文件访问必须经过业务权限校验。
- API 必须配置明确的跨域允许列表、安全响应头和统一错误处理；生产环境不得向客户端返回堆栈、SQL、文件路径、内部账号或其他实现细节。
- 对登录、验证码、密码重置、审批、文件上传、批量查询等敏感或高消耗接口，应根据风险增加频率限制、重复提交控制、审计日志和异常告警。
- 安全修复必须补充能够复现问题的测试，并验证修复后攻击路径不可用、正常业务仍然可用。
- 完成前后端开发前，至少验证依赖漏洞、认证与越权、输入校验、SQL 注入、XSS、CSRF、文件上传、敏感信息泄露和错误信息暴露等风险；不得仅以“功能可以运行”作为完成标准。
